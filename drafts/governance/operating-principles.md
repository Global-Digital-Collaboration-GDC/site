# Operating Principles

## Working assumptions

The organizer hub should remain:

- practical rather than ceremonial
- transparent by default
- friendly to asynchronous collaboration
- easy for non-technical contributors to update

## Draft principles

### Clarity over volume

Capture only what the team needs to make progress: decisions, owners, dates, dependencies, and open questions.

### Public when possible

Because the repository is public, content should avoid sensitive operational details while still making the work legible to partners and contributors.

### Reuse one source of truth

Formal content belongs in this site. Brainstorms, scratch notes, and evolving checklists can live in the GitHub wiki until they mature.

### Time-box task forces

Task forces should have a clear objective, a lead, and an expected review date.
